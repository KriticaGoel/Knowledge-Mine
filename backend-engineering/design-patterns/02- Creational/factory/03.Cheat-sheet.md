# 🧠 Factory Design Pattern — Cheat Sheet

## 🎯 Core Idea

<details>
<summary>Show / Hide</summary>

Object creation is separated from business logic.

```
Client → Factory → Object
```

</details>

---

## ⚠️ When to Use Factory

<details>
<summary>Show / Hide</summary>

* Multiple implementations exist
* Object creation varies at runtime
* You see giant if-else / switch
* Complex object creation logic

👉 Rule:
If creation changes → Use Factory

</details>

---

## ❌ Without Factory (Problem)

<details>
<summary>Show / Hide</summary>

```java
if(type.equals("CSV")) return new CsvParser();
else if(type.equals("JSON")) return new JsonParser();
```

### Issues:

* Violates OCP
* Tight coupling
* Hard to test
* Duplicate creation logic

</details>

---

## 🏗️ Simple Factory

<details>
<summary>Show / Hide</summary>

### Idea

Central class creates objects

```java
Parser parser = ParserFactory.getParser(type);
```

### Pros

* Centralized creation
* Removes `new` from client

### Cons

* Still modifies factory for new types

</details>

---

## 🧱 Factory Method

<details>
<summary>Show / Hide</summary>

### Idea

Subclass decides object creation

```
Abstract Creator → Concrete Creator
```

### Key Method

```java
createParser()
```

### Pros

* No if-else
* Follows OCP

### Cons

* More classes

</details>

---

## 🧩 Abstract Factory

<details>
<summary>Show / Hide</summary>

### Idea

Factory of related objects

Example:

* Button + Checkbox (Android/iOS)

### Benefit

* Ensures compatible families

</details>

---

## 🔥 Spring + Factory

<details>
<summary>Show / Hide</summary>

### Spring

Creates objects (beans)

### Factory

Selects correct object at runtime

```
Spring → creates all parsers
Factory → chooses one
```

</details>

---

## 🧠 Final Memory Hook

<details>
<summary>Show / Hide</summary>

* Simple Factory → Central switch
* Factory Method → Subclass switch
* Abstract Factory → Family switch
* Spring + Factory → Production pattern

</details>
